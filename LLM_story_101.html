01
Where It All Begins
The Problem — Machines Cannot Read

Computers do not understand language. They understand one thing only: arithmetic on numbers. Add, multiply, compare — that is the complete vocabulary of a CPU or GPU. Text, in its raw form, means absolutely nothing to a machine.

So the central challenge of building a language model is this: how do you take a sentence like "The king sat on his throne" and transform it into numbers in a way that actually preserves meaning? Not just arbitrary numbers, but numbers that somehow know "king" and "queen" are related, that "Paris" is a capital city, that the word "bank" means something different next to "river" than next to "deposit"?

The answer is a pipeline — a carefully designed sequence of transformations. This document is the complete story of that pipeline, told in the order that makes each step feel inevitable once you understand the one before it.

The Central Insight

Every capability of a modern LLM — answering questions, writing code, reasoning through problems — flows from one underlying achievement: converting text into numbers in a way that encodes meaning geometrically. Similar meanings end up close together in mathematical space. That geometric closeness is what makes everything else possible.

02
Step One of the Pipeline
Tokens — The Alphabet of LLMs

Before anything else can happen, raw text must be broken into discrete units the model can work with. These units are called tokens. Tokens are not words — they are sub-word units, syllable-sized chunks that sit between individual characters and full words.

"I love ice cream"   →  ["I", " love", " ice", " cream"]      // 4 tokens
"unbelievable"       →  ["un", "believ", "able"]               // 3 tokens
"ChatGPT"            →  ["Chat", "G", "PT"]                    // 3 tokens
"import numpy"       →  ["import", " numpy"]                   // 2 tokens

Once the text is broken into tokens, each token is assigned a unique integer ID from a fixed vocabulary. "king" might become token ID 4571. "queen" might become 7542. These IDs are the raw material passed into the model.

PropertyTypical Value
Approximate size1 token ≈ 0.75 words in English
Vocabulary size50,000 – 100,000 unique tokens per model
What "128k context window" meansThe model can process 128,000 tokens at once (~96,000 words)
Key Question — What Is the Token ID Actually Used For?

The token ID is purely a lookup key. No math is ever done on the number 4571 itself. It works exactly like a primary key in a database: it tells you which row to retrieve, but the number itself encodes nothing about the content. The token ID is just a pointer to the next step.

Mental Model

Tokens are the alphabet of LLMs. The model never sees raw text — only a sequence of integer IDs. All language is converted into a list of numbers before the model processes a single character of meaning.

03
What a Model Is Made Of
Weights and Parameters — The Building Blocks

A model is, at its core, a very large collection of floating-point numbers called weights (also called parameters — the two words mean the same thing). A single weight is nothing more than a decimal number. What makes weights powerful is that the model uses billions of them simultaneously during every computation.

// Simplified: a sentiment detector using weights
score = (0.8 × "love") + (0.9 × "great") + (-0.7 × "terrible")
//        ↑ weight           ↑ weight               ↑ weight

// Before training — weights are random, meaningless:
score = (0.1 × "love") + (0.1 × "great") + (0.1 × "terrible")

// After training — weights reflect reality:
score = (0.8 × "love") + (0.9 × "great") + (-0.7 × "terrible")

Training is the process of nudging these numbers from random to meaningful. The entire intelligence of an LLM lives in these weight values.

Where Weights Live — Inside a Transformer Layer

The Chinchilla Law

In 2022, DeepMind showed that model size and training data volume should scale together. A smaller model trained longer often beats a bigger model trained shorter.

// Chinchilla optimal ratios (DeepMind, 2022)
 10B parameter model  →  optimal training data: ~200B  tokens
 70B parameter model  →  optimal training data: ~1.4T  tokens
175B parameter model  →  optimal training data: ~3.5T  tokens
Mental Model

When you hear "Llama 3 is a 70B model", it means there are 70 billion floating-point numbers stored in a file on disk. That file is the model — all its knowledge and capability encoded in those numbers. When you run the model, your input flows through all of them.

04
Training — Phase 1 of 3
Pre-training — Where All Knowledge Comes From

With the architecture defined and weights initialized to random values, training begins. Pre-training is the most expensive, most time-consuming, and most consequential phase. It is where the model acquires virtually all of its knowledge. And the task is devastatingly simple:

"Given everything before this point, predict the next token. No chat. No instructions. No helpfulness. Just: what comes next?"

This task runs trillions of times across an enormous corpus: Wikipedia, books, GitHub, scientific papers, news, web pages. The model makes a prediction, measures how wrong it was, and adjusts its weights slightly. Billions of times.

// The training task — repeated trillions of times
"The capital of France is ___"         →  Paris
"She opened the door and ___"          →  walked / saw / found
"import numpy ___"                     →  as
"The king sat on his ___"              →  throne

Because the model must predict text from every domain, it is forced to learn everything that helps it predict accurately: grammar, facts, reasoning, code, logic. Nobody teaches these explicitly. They emerge as side effects of getting good at next-token prediction.

The Result — Brilliant but Feral

At the end of pre-training you have a base model. Extraordinarily knowledgeable, but it has no concept of being an assistant. Ask it something and it continues the text pattern:

You:        "What is the capital of France?"
Base model: "What is the capital of Germany? What is the capital
             of Spain? What is the capital of Italy? ..."

It is not being difficult — it learned that this type of question is usually followed by more of the same. Base models are never shipped to users.

Pre-training: massive corpus → next-token prediction → repeated weight adjustments → base model
05
Training — Phase 2 of 3
Fine-tuning — Teaching It to Be Helpful

The base model knows an extraordinary amount but cannot have a conversation. Fine-tuning trains it on a smaller, curated dataset of human-written question-and-answer pairs. Same task — predict next token — but now every example follows a conversational shape:

// Fine-tuning data format
[Human]:     How do I reverse a string in Python?
[Assistant]: You can use slicing: s[::-1]
             "hello"[::-1] gives "olleh"

[Human]:     What causes rain?
[Assistant]: Rain forms when water vapor in clouds condenses
             into droplets heavy enough to fall.

Fine-tuning does not add new knowledge — all knowledge was already absorbed during pre-training. It only teaches a new behaviour on top of existing knowledge. And it is far cheaper: days instead of months.

06
Training — Phase 3 of 3
RLHF — Teaching It Values

Fine-tuning taught the model to talk. RLHF — Reinforcement Learning from Human Feedback — teaches it to be good. This is what separates a chatbot from an AI assistant like Claude.

RLHF four-step process: human preference rankings → reward model → weight update → aligned assistant
The Three Phases Together

Pre-training builds the knowledge and intelligence — the model absorbs the world.
Fine-tuning teaches the format and behaviour — the model learns to converse.
RLHF instils values and preferences — the model learns what "good" means.

Chat is not the fundamental task. It is a behaviour layered on top of a far more fundamental capability: predicting what comes next from a vast understanding of human language and thought.

07
From Tokens to Meaning
The Embedding Lookup Table

Tokenization gives us integer IDs — but an ID like 4571 is still just a number. It says nothing about what "king" means or how it relates to "queen". The embedding lookup table bridges this gap.

The table is a large matrix — one row per token in the vocabulary. Each row is a long list of floating-point numbers. This list is the token's embedding vector, and it is where meaning begins to live.

// Embedding table (simplified — real: ~50,000 rows × 768–3072 cols)
Token ID   →  Embedding Vector

4571 ("king")  →  [0.21, -0.45, 0.87, 0.33, -0.12, ...]  ← 1536 floats
7542 ("queen") →  [0.19, -0.42, 0.91, 0.31, -0.09, ...]  ← very similar!
1234 ("car")   →  [-0.80, 0.12, -0.55, 0.67, 0.44, ...]  ← very different
Key Question — Is the Token ID Used in Creating the Embedding?

Yes — but only as a row index, not as a number to compute with. The model does no arithmetic on 4571. It uses it purely as an address: "give me row 4571 from the table." If "king" were reassigned to token ID 99999 tomorrow, its embedding vector would be completely unchanged. The ID is arbitrary — what matters is the vector it points to.

Token ID is just a row index. The row's contents — the vector — is what carries meaning.
08
Inside the Vector
Dimensions — What They Actually Are

The embedding vector for "king" has 1536 numbers. What does each one represent? This is one of the most important questions to get right, because the intuitive answer is wrong.

The dimensions are not human-designed features. Nobody labelled them. No dimension #5 means "royalty-ness". Nobody — not even the model's creators — knows what any specific dimension means.

Real Dimension Counts Across Models

1536 is not a standard. It was OpenAI's choice for one model, and it became widely cited because that model was popular. Every model has its own count.

How the Number Is Chosen

The dimension count flows from the architecture. In a Transformer, embedding dimension = attention heads × head size:

BERT-base:   12 heads × 64 dims each  =  768  dimensions
BERT-large:  16 heads × 64 dims each  =  1024 dimensions
GPT-3:       96 heads × 128 dims each =  12288 dimensions

Teams then benchmark different sizes on the MTEB (Massive Text Embedding Benchmark) and choose the best quality-to-cost tradeoff.

Matryoshka Representation Learning (MRL)

OpenAI's text-embedding-3 models use a technique where the first N dimensions are useful on their own — like nesting dolls. You can truncate the vector to save cost without retraining:

text-embedding-3-large, full  3072 dims  →  highest accuracy
text-embedding-3-large, first 1024 dims  →  good quality, cheaper
text-embedding-3-large, first  256 dims  →  fast and cheap, still useful
Critical Rule

Vectors from different models are completely incompatible. OpenAI's dimension #42 has zero relationship to BERT's dimension #42. If you embed your documents with model A, you must embed your queries with model A too. You cannot mix models in the same vector database.

09
The Deep Truth
Learned, Abstract, Distributed — What That Really Means

No one decided what the dimensions mean. The model invented its own representational language from scratch, and even its creators do not fully understand it.

How Features Emerge Without Being Designed

During pre-training the model's only task is predicting the next token. No human labels anything. The model just sees billions of sentences and notices patterns: "king", "queen", "emperor", "pharaoh" appear in very similar contexts. "Paris", "London", "Tokyo" all behave like capital cities. It discovers all of this because noticing patterns helps it predict better — not because it was told to.

Why "Distributed" Is the Key Word

These patterns are not stored in single dimensions. They are spread across many dimensions simultaneously — like a chord rather than a single note:

// "king" and "queen" share a pattern across multiple dimensions
// Similar on some (authority-like), different on others (gender-like)
// But NO single dimension cleanly means "authority" or "gender"

"king"   →  dim[5]=0.8,  dim[47]=-0.3, dim[312]=0.9, dim[891]=0.2
"queen"  →  dim[5]=0.8,  dim[47]=0.3,  dim[312]=0.9, dim[891]=-0.2
"car"    →  dim[5]=-0.4, dim[47]=0.1,  dim[312]=-0.7, dim[891]=0.0
The Word2Vec Discovery (2013)

vector("king") − vector("man") + vector("woman") ≈ vector("queen")

Nobody programmed this. The model discovered a consistent geometric "gender direction" in its vector space — and that royalty words are offset along it. Pure emergence from next-token prediction.

Retrain the Same Model — Get Different Dimensions

Train the exact same architecture on the exact same data twice, changing only the random initialization seed. Both models perform identically — but encode everything in completely different dimensions. In one run, the "gender direction" lives in dims [5, 47]. In another, it lives in [8, 201]. The geometry is preserved. The specific axes are not. What matters is relationships between vectors, not which axis carries which pattern.

The Field Trying to Fix This — Mechanistic Interpretability

Anthropic and other labs invest heavily in reverse-engineering what individual neurons and weight patterns mean. So far researchers have found:

FoundA neuron that activates specifically for text inside quotation marks

One specific neuron in GPT-2 fires consistently whenever the model encounters text between quote marks — regardless of the content.

FoundClusters responding to DNA sequence patterns

Neurons that activate for sequences of A/T/G/C nucleotide letters — despite the model never being told what DNA is.

FoundA direction encoding "about to refuse a request"

A direction in activation space that consistently corresponds to when the model is going to decline to answer something.

But for every neuron interpreted, thousands remain unexplained. It is like archaeology — excavating a structure built by no conscious architect.

"Humans built the training loop. Data shaped the weights. Weights created the dimensions. Dimensions encode meaning. But no human chose what meaning to encode or how."
10
The Critical Distinction
Static vs Contextual Embeddings

The embedding lookup table gives every token a fixed vector. So does "bank" always get the same vector? If so, how does the model know whether it means a financial institution or a riverbank?

Type 1 — Static Embeddings (The Lookup Table)

At the input layer, yes — every token has a fixed initial vector. But a single fixed vector cannot capture multiple meanings. This was the fundamental limitation of early models like Word2Vec (2013) and GloVe (2018).

// Static: same vector regardless of context — ambiguity ignored
"bank"  →  token ID: 4521  →  [0.21, -0.45, 0.87, ...]  ← always identical
// whether "deposit money" or "river bank" — same vector

Type 2 — Contextual Embeddings (The Transformer Revolution)

Modern LLMs solve this with the Transformer's attention mechanism. The static lookup vector is just the starting point. It then flows through many Transformer layers. At each layer, every token examines every other token in the sequence and reshapes its own vector based on surrounding context.

By the time the vector exits the final layer, it has been completely transformed. Same word, same starting vector, entirely different final vector depending on the sentence:

// Contextual: same start, different finish based on context

"I went to the bank to deposit money"
    initial:  [0.21, -0.45, 0.87, ...]   ← same starting vector
    after 96 layers: [0.12, 0.87, -0.34, ...]  ← financial institution

"I sat by the river bank and fished"
    initial:  [0.21, -0.45, 0.87, ...]   ← same starting vector
    after 96 layers: [0.91, -0.23, 0.67, ...]  ← riverbank

BERT (2018) was the breakthrough that introduced bidirectional contextual embeddings. It immediately set new records on virtually every NLP benchmark.

Static models give one vector per word. Contextual models reshape the vector based on every surrounding word.
11
When Does It All Run?
Training vs Inference — The Complete Picture

Does all of this — the contextual reshaping, the attention mechanism, the vector transformations — happen during training, or when you actually use the model?

Both. But in fundamentally different ways.

During Training — Learning How to Reshape

During pre-training, contextual reshaping absolutely runs — it must, so the model can produce predictions, measure how wrong they are, and adjust weights. But the purpose is to learn what the Transformer layer weight matrices should contain. The model is figuring out: "given this context, how should I reshape these vectors so that predictions improve?"

When training ends, those weight matrices are frozen. They now encode the model's learned understanding of how context should modify meaning.

During Inference — Actually Doing the Computation

When you send a message to Claude right now, the entire pipeline runs live on your input, using those frozen weights. Tokenization, embedding lookup, all contextual reshaping across every Transformer layer — all of it happens in real time, fresh for your specific input.

// Every message you send triggers the full pipeline
"What is the capital of France?"
   ↓  Tokenizer                // splits into token IDs
   ↓  Embedding lookup          // fetches initial vectors — frozen weights
   ↓  Transformer layer 1       // attention reshapes vectors — frozen weights
   ↓  Transformer layer 2 … 96  // more reshaping, all frozen
   ↓  Output layer              // probability distribution over vocabulary
   ↓  "Paris"
The pipeline is identical in both phases. Training updates weights; inference uses them read-only.
AspectTrainingInference
Embedding lookupRuns — values being learnedRuns — values frozen, just fetched

The Calculator Analogy

Training = designing and building the calculator — figuring out all the circuits and logic.
Inference = pressing buttons and getting results — the circuits are fixed, but computation happens fresh every keypress.

The calculator does not relearn arithmetic each time you use it. But it absolutely computes every time you press a button. And a longer calculation costs more keypresses — which is why longer conversations cost more to run.

12
The Geometry of Meaning
Vectors — Points in Meaning-Space

We have used the word "vector" throughout this document. Let us now be precise. A vector is simply an ordered list of numbers — a point in N-dimensional space. The embedding vector for "king" is a point in 1536-dimensional space. Because of how the model was trained, similar meanings end up at nearby points in that space.

Embedding = the process of converting a word to a vector
Vector = the result — the actual list of numbers, the point in space

"king" → [0.21, -0.45, 0.87, ...] ← a point in 1536-D space
"queen" → [0.19, -0.42, 0.91, ...] ← nearby point — similar meaning
"car" → [-0.80, 0.12, -0.55, ...] ← far away — unrelated meaning

Meaning is geometric. Related concepts cluster together. And meaningful relationships can be computed as vector arithmetic:

vector("king") − vector("man") + vector("woman") ≈ vector("queen")
// The "gender direction" is consistent enough to do arithmetic on.
// Nobody programmed this. Pure emergence from training.
Mental Model

If embedding is the translation process, the vector is the resulting address in meaning-space. Every piece of text has an address. Addresses that are close together represent things that mean similar things.

13
Measuring Meaning
Cosine Similarity and Vector Databases

Now that text is represented as points in space, we can measure how similar two pieces of text are by measuring how close their vectors are. But "close" measured how?

Why Not Euclidean Distance?

Straight-line distance is sensitive to the magnitude (length) of vectors. Two vectors pointing in the same direction but of different lengths would appear "dissimilar" — even though they encode the same meaning. Magnitude is irrelevant to meaning.

Cosine Similarity — Angle Only

Cosine similarity measures only the angle between two vectors, ignoring their length entirely. Vectors pointing the same way score 1.0. Vectors at right angles score 0.0. Opposite directions score −1.0.

cosine_similarity("king",  "queen")   →  0.92   // very similar meaning
cosine_similarity("king",  "car")     →  0.15   // unrelated
cosine_similarity("hot",   "cold")    → -0.30   // somewhat opposite
cosine_similarity("Paris", "London")  →  0.88   // both capital cities
ScoreMeaning
1.0Identical direction — semantically identical
0.8 – 1.0Very similar meaning
0.5 – 0.8Related concepts
0.0 – 0.5Loosely related or unrelated
Below 0Somewhat opposite in meaning

How Vector Databases Work

At index time (loading documents into the DB)
Each document → embed with model A → store vector + metadata

At query time (searching)
User query → embed with same model A → get query vector
Compare against all stored vectors using cosine similarity
Return top-K most similar documents

Result: semantically relevant docs — even with no shared keywords 
// Semantic search — no shared keywords needed
query: "affordable notebook computers"  →  embed  →  query vector

"cheap laptop deals"        →  similarity: 0.91  ✓ returned
"budget ultrabook review"   →  similarity: 0.87  ✓ returned
"recipes for pasta"         →  similarity: 0.08  ✗ filtered out

Mental Model

Cosine similarity asks: are these two vectors pointing in roughly the same direction? It does not matter how long they are — only where they aim. If they point the same way, the text has similar meaning. This is how "affordable notebook" finds "cheap laptop" with zero shared keywords.
